1. Elevator pitch

AsphaltGuard is a web dashboard for monitoring road/asphalt condition using data from an RC car or field rig: GPS position, thermal camera readings, visible-light captures, and YOLO-based crack/pothole detection. Operators sign in, view live video feeds, inspect historical captures on a map, and see aggregate stats (defect counts, thermal condition bands).

“It’s a React SPA backed by Supabase for auth and persistence, with real-time telemetry polled from our streaming server at stream.asphaltguard.online.”

4. App architecture

  • SPA: Single-page app; all routes client-side. Netlify/Render rewrite /* → index.html for deep links.
  • Entry: main.tsx wraps App in BrowserRouter.
  • Global side effect: LastLoginTracker runs on every page — listens to Supabase auth and updates users.last_login.
  • Dashboard v2 shell: DashboardShell provides sidebar (SideNavigation), page title, fade transitions on navigate, loading overlay.
  • Map view: Standalone full-screen layout (not using DashboardShell); auth-guard redirects to /login.

5. Authentication & users

  • Supabase Auth: signInWithPassword, signUp, signOut, getSession, onAuthStateChange.
  • Profile table: On signup, a row is inserted into users with profile fields linked by user_id (auth UUID).
  • Last login: LastLoginTracker + updateLastLogin() write ISO timestamp to users.last_login.
  • Protected UX: Map explicitly checks session; other dashboard pages load user data but may not hard-redirect (worth noting in security discussions).

User fields referenced in UI: username, full_name, email, role, contact_number, created_at, user_id.

6. Supabase data model

No migration files live in this repo — schema is managed in Supabase cloud. Tables and buckets used in code:

Table: captures primary

Field inspection records from the RC rig (v2 pipeline).

  • id, captured_at, gps_latitude, gps_longitude, gps_valid
  • Thermal: thermal_max_c, thermal_min_c, thermal_mean_c, thermal_ambient_c, thermal_grid (2D array)
  • Vision: image_url, yolo_crack, yolo_pothole
  • Vehicle: rccar_temperature_c

Queries typically filter gps_valid = true and order by captured_at descending.

Table: users

App profiles tied to Supabase Auth IDs.

Table: scan_history v1 / demo

Older pothole-scan workflow: original_img_url, annotated_img_url, pothole_count, created_at.

Storage bucket: ai-pothole-scanner

Used by demo/YOLO26.tsx for original + annotated images; public URLs stored in scan_history.

8. Features by screen

Dashboard (/dashboard)

  • Loads signed-in user from users.
  • Aggregates all valid GPS captures: total inspected, defect count (crack + pothole flags), thermal condition histogram (Good / Fair / Deteriorating / Critical).
  • Shows latest capture time (Philippine timezone), peak and average max thermal.

Live Feed (/live-feed)

  • Two <img> streams: visible camera + thermal camera.
  • Status handling: loading / online / error with auto-retry every 3s.

Map (/map)

  • Centered near PSU coordinates; markers for captures colored by defects / thermal thresholds.
  • Live drone/RC GPS marker when /gps reports valid fix; stale vs active styling.
  • Side panel: “Asphalt Condition” — capture list, detail view with thermal grid (ThermalGrid inferno colormap).
  • Browser geolocation for “fly to user”.

History (/history)

  • Same captures data as map list, optimized for browse/detail (not map-centric).
  • Components: HistoryCaptureList, HistoryCaptureDetail.

Account (/account)

  • Displays profile + auth email; logout → /login.

Users (/users)

  • Lists all rows from users (name, username, email, role).

AI Models (/ai-models)

  • Static in-app catalog: “AsphaltGuard Model v1” (YOLOv26n, F1 0.81) and v2 placeholder.
  • No live training pipeline in this repo — presentation layer only.

YOLO demo (/demo/yolo26)

  • User uploads image → HF detect API → uploads to Supabase Storage → inserts scan_history.

Legacy dashboard v1

  • /dashboard-v1 — stats from scan_history (pothole counts, thermal fields if present).
  • /dashboard-v1/scanhistory — list of scans; ScanHistoryCard component.
  • Scan.tsx is a stub (“DashboardScan”).

10. Domain logic (thermal & defects)

Centralized in captureUtils.ts and mirrored in dashboard/map:

Max surface temp (°C)Condition
≤ 50Good
51–60Fair
61–70Deteriorating
> 70Critical
  • Defects: yolo_crack and yolo_pothole booleans increment defect counts; map pins use alert styling when true.
  • Time labels: Captures titled Morning / Afternoon / Evening / Night based on Asia/Manila hour.
  • RC temperature: Prefers rccar_temperature_c, falls back to thermal_mean_c.

11. Deployment & configuration

  • Build output: dist/ static files after npm run build.
  • Netlify: SPA fallback /* → /index.html (status 200).
  • Render: Same rewrite pattern in render.yaml.
  • Secrets: .env is gitignored; must set Supabase URL and anon/publishable key for any environment.
  • Dev server: vite --host exposes LAN for mobile/tablet testing.

12. Development workflow

  1. Clone repo → npm install
  2. Create .env with VITE_SUPABASE_URL and VITE_SUPABASE_PUBLISHABLE_DEFAULT_KEY
  3. npm run dev → open local URL (default Vite port)
  4. npm run lint before commits
  5. npm run build to verify production compile (TypeScript project references)

What’s not in this folder: Supabase SQL migrations, edge functions, Python/embedded RC car firmware, stream server implementation, or model training code — those are separate systems that feed this UI.

13. Discussion prompts

Use these when presenting or defending design choices:

Product

  • Who is the primary user — field operator, supervisor, or public works?
  • How do thermal bands map to maintenance decisions?
  • What is the capture → upload → dashboard latency target?

Architecture

  • Why Supabase vs custom API?
  • Why poll stream APIs every 2s instead of WebSockets?
  • v1 scan_history vs v2 captures migration plan?

Security

  • RLS policies on captures and users?
  • Should all routes enforce auth like /map?
  • Are storage buckets public appropriately?

ML / IoT

  • Where does YOLO run on the vehicle vs cloud?
  • How is thermal_grid produced and stored?
  • Accuracy and calibration of GPS + thermal alignment?

One-paragraph summary (read aloud)

AsphaltGuard is a Vite-powered React dashboard for an asphalt inspection platform. It authenticates users with Supabase, stores rich capture records (GPS, thermal stats, YOLO flags, images), and visualizes them on a Leaflet map and history views. Live camera and telemetry come from a dedicated streaming host. The codebase includes a legacy pothole-upload demo (Hugging Face + Storage) and a newer v2 experience built around the captures table, with deployment configured for static hosting on Netlify or Render.
