1. One-minute overview

This repo is an edge inspection platform on a Raspberry Pi for asphalt / road monitoring (“AsphaltGuard”). It combines:

  • Visible camera + YOLO11 segmentation — detects cracks and potholes in real time.
  • MLX90640 thermal camera — 32×24 temperature grid over I2C for surface heat patterns.
  • GPS (NEO-6M) — geotags readings via UART / NMEA.
  • RC car probe (ESP32 + MAX6675) — Bluetooth temperature on a mobile test rig.
  • Optional 433 MHz RF — RadioHead ASK decoder via pigpio.
  • FastAPI server — dual MJPEG streams, stats API, local captures, upload to Supabase.
How to introduce it
  • “Sensors run as small C++ daemons that write JSON to /tmp; Python reads those files and serves a browser dashboard.”
  • “Inference uses an NCNN-exported YOLO model on the Pi for speed; PyTorch .pt is also available.”
  • “Captures bundle camera frame + thermal grid + GPS + RC temperature into CSV, then optionally sync to Supabase.”

2. Architecture & data flow

The design is deliberately loosely coupled: C++ processes own hardware; Python polls JSON sidecars.

┌─────────────────────────────────────────────────────────────────────────┐ │ Raspberry Pi (Linux) │ ├─────────────────────────────────────────────────────────────────────────┤ │ C++ daemons (CMake / build/) │ │ thermal_camera ──► /tmp/thermal_camera_latest.json │ │ gps_reader ──► /tmp/gps_latest.json │ │ rccar_reader ──► /tmp/rccar_latest.json (Bluetooth RFCOMM) │ │ rf_receiver ──► stdout (433 MHz, pigpio) │ ├─────────────────────────────────────────────────────────────────────────┤ │ scripts/server.py (FastAPI + uvicorn) │ │ • Picamera2 + YOLO (NCNN or .pt) → MJPEG /video/camera │ │ • Poll thermal JSON → MJPEG /video/thermal │ │ • Poll GPS / RC JSON → /gps, /rccar, /api/stats │ │ • POST /capture → captures/*.jpg + captures.csv │ │ • POST /upload → Supabase Storage + Postgres │ └─────────────────────────────────────────────────────────────────────────┘ ▲ ▲ ▲ │ I2C │ UART │ Bluetooth SPP MLX90640 NEO-6M GPS ESP32 "AsphaltGuard-RC" Pi Camera (HW-248) MAX6675 thermocouple
Design choices worth mentioning
  • Atomic JSON writes (write temp file, rename) so readers never see half-written data.
  • Same pattern lets you run C++ readers under systemd while developing the Python server separately.
  • Default entry point: ./run → .venv/bin/python scripts/server.py --conf 0.10 --imgsz 224 --stride 5 --ncnn

ESP32 + MAX6675 + motor driver Bluetooth SPP → /dev/rfcomm0 rccar_reader, esp32/rccar.ino BT name AsphaltGuard-RC; send TEMP for reading
433 MHz ASK RF module GPIO 13, pigpio rf_receiver RadioHead RH_ASK @ 2000 bps; pairs with Arduino/ESP32 TX

Pi configuration (from README)

  • I2C: dtparam=i2c_arm=on, optional dtparam=i2c1_baudrate=400000
  • UART: enable_uart=1, often dtoverlay=disable-bt; remove serial console from cmdline
  • User in i2c and dialout groups for sensor access

4. C++ programs (src/)

Built with CMake (CMakeLists.txt) or legacy Makefile. Targets install to build/.

BinarySourceLinked libsRole
thermal_camera src/thermal_camera_reader.cpp mlx90640, pthread Reads frames, optional terminal heatmap, exports JSON with full temp grid
gps_reader src/gps_reader.cpp — Parses NMEA ($GPGGA, $GPRMC), exports fix JSON
rccar_reader src/rccar_reader.cpp pthread Polls ESP32 over RFCOMM; reconnect logic for power cycles
rf_receiver src/rf_receiver.cpp pigpio, rt, pthread Decodes RH_ASK packets; not wired into server yet

Third-party C++ library

  • lib/mlx90640/ — Pimoroni MLX90640 port (Apache 2.0 / Melexis), Linux I2C driver

Typical JSON sidecars

/tmp/thermal_camera_latest.json  →  width, height, ambient_c, min/max/mean, temperatures[][]
/tmp/gps_latest.json               →  valid, lat, lon, altitude, satellites, utc_time, …
/tmp/rccar_latest.json             →  connected, valid, temperature_c, error_code, …
Talking points
  • C++ keeps timing-sensitive I2C/UART/Bluetooth off the Python GIL.
  • rccar_reader can re-bind RFCOMM if the ESP32 drops — important for field demos.

6. ML models (model/)

  • Task: YOLO11n segmentation — classes crack (0), pothole (1)
  • Training source: Ultralytics; dataset refs in metadata (all-together-2-1, etc.)
  • Production on Pi: NCNN export in best (2)_ncnn_model/ (224×224, stride 32)
  • Alternate: dual_ncnn_model/ (640×640 imgsz in metadata)
  • PyTorch weights: best (2).pt (referenced in code; use --ncnn flag to prefer NCNN)

Server flags: --conf, --imgsz, --stride (infer every N frames), --ncnn, --max-det.

Talking points
  • NCNN reduces latency on ARM; stride > 1 trades accuracy refresh rate for FPS.
  • Detection booleans yolo_crack / yolo_pothole are stored per capture row.

7. FastAPI web server (scripts/server.py)

Background threads

  • camera_thread — Picamera2 capture + YOLO overlay → JPEG buffer
  • thermal_thread — polls thermal JSON → colormap JPEG
  • gps_thread — polls GPS JSON (unless --no-gps)
  • rccar_thread — polls RC JSON (unless --no-rccar)

HTTP routes

RouteDescription
GET /Embedded dashboard (dual video + GPS + RC bars + Capture/Upload buttons)

GET /video/thermal, /thermalMJPEG thermal heatmap

8. Supabase (cloud)

  • Storage bucket: captures (public read; anon insert/update per migration)
  • Table: public.captures — timestamps, image URL, camera stats, thermal summary + thermal_grid JSONB, GPS fields, RC car fields, YOLO flags
  • Migrations: supabase/migrations/001_captures.sql, 002_add_rccar_columns.sql
  • Config: SUPABASE_URL, SUPABASE_KEY, SUPABASE_BUCKET env vars (defaults exist in server.py — override in production)
Talking points
  • Edge-first: CSV + images on disk, then batch upload when network is available.
  • RLS policies allow anonymous insert for demo kits; tighten for production.

9. ESP32 RC car firmware (esp32/rccar.ino)

  • Bluetooth: BluetoothSerial, device name AsphaltGuard-RC
  • Temperature: MAX6675 on SPI pins (CS 5, SO 19, CLK 18); responds to T / TEMP
  • RC control: CH1/CH2 PWM-style inputs (GPIO 32/35), motor H-bridge outputs, auto-stop after 150 ms idle
  • Flashed with: Arduino IDE / PlatformIO (Bluetooth enabled in sdkconfig)

Pi pairs once via bluetoothctl; scripts/rccar-rfcomm-bind.sh binds MAC → /dev/rfcomm0.

10. 433 MHz RF receiver

rf_receiver uses pigpio for microsecond edge timing and decodes RadioHead RH_ASK at 2000 bps on GPIO 13. Useful for wireless triggers or remote sensors; currently a standalone diagnostic tool, not integrated into server.py.

Build dependency: libpigpio-dev, pigpio daemon running.

15. Discussion prompts

Use these when presenting to technical or non-technical audiences:

Problem & value

  • What road defects are we trying to catch early, and why both visible AI and thermal data?
  • How does GPS + timestamp prove where a defect was seen during a drive?

Architecture

  • Why split sensor readers (C++) from the web app (Python) instead of one monolith?
  • What breaks if thermal_camera stops but server.py keeps running?

ML / performance

  • Why NCNN on the Pi instead of full PyTorch?
  • What does --stride 5 mean for detection latency vs CPU load?
  • How would you retrain or swap models (export_ncnn.py, new model/ folder)?

Field hardware

  • Walk through RC car probe: ESP32 → Bluetooth → Pi → dashboard.
  • What is the RF receiver for, and could it trigger captures?

Data & cloud

  • What is stored locally vs in Supabase per capture row?
  • How would you secure anon keys and RLS before a public pilot?

Operations

  • Which systemd units must be enabled for a “headless” demo box?
  • How do you verify I2C (i2cdetect -y 1) and GPS (raw NMEA) before the full stack?
